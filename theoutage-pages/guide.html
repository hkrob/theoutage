TheOutage

Guide

How TheOutage works, and why each field on a submission exists.

What this is

TheOutage is a public, community-maintained record of major outages — power, internet, cloud, transport, water, telecom, financial, healthcare, government, and other infrastructure incidents. Anyone can submit a report; every submission is checked by a moderator before it goes live, so the published feed stays accurate and free of spam.

The submission workflow

Every outage report moves through a small set of states:

  • Draft — saved but not submitted. Only you can see it. Edit or delete it freely.
  • Pending review — submitted, waiting for a moderator. Visible to you and to moderators/admins, not to the public.
  • Published — approved and live on the public feed.
  • Rejected — a moderator sent it back with a reason (shown on the outage page). Edit and resubmit any time.

Editing a published outage always sends it back to "pending review" — any change, however small, gets a fresh moderation pass before it's public again. This keeps the public record trustworthy: nothing changes on the live feed without a moderator seeing it first.

Once an outage has ever been submitted for review, it can't be deleted outright (it's part of the historical record) — only unsubmitted drafts can be removed completely.

Fields, and why they're there

Title

A short, specific headline — what happened and roughly where. This is what shows up in search results and the feed.

Entity

The company, organisation, or department actually responsible for the affected system — e.g. the utility, cloud provider, airline, or government agency. This is what makes the record searchable by "who," not just "what," and is separate from the Title so it stays consistent even when headlines are worded differently.

Stock code (optional)

The ticker symbol for the impacted company, if it's publicly traded (e.g. AAPL). When set, the outage page shows a live price next to it. Left blank for entities that aren't publicly traded — most government agencies, utilities, and smaller organisations. Live pricing currently only works for US-listed tickers — international exchanges (ASX, LSE, and others) aren't available on the free data plans this site uses yet, so the code will still be recorded but no price will show.

Description

The full account: what happened, the scope of impact, and the cause if known. This is what gets full-text searched.

Category

The kind of infrastructure affected (power, internet, cloud, transport, water, telecom, financial, healthcare, government, other) — the main way the feed gets filtered.

Severity

P1 High, P2 Medium, or P3 Low — how serious the outage is, independent of category. The naming follows standard incident-priority convention (P1 = most severe).

Current status

Where the incident itself stands right now — Investigating, Identified, Monitoring, or Resolved. This is deliberately separate from the moderation status above: an outage can be "Published" (moderators have approved the report as accurate) while its current status is still "Investigating" (the underlying incident isn't over). Update this as an outage develops — note that, like any edit to a published outage, changing it sends the report back for a quick re-review.

Tags (optional)

Free-text, comma-separated keywords for anything the fixed fields don't capture (e.g. a specific cause like "transformer" or "heatwave").

Country / City

Where the outage happened. Country is a fixed list for consistent filtering; city is free text and optional since not every outage is city-scoped.

Started / Ended

When the incident began, and when it ended — leave "Ended" blank and check "still ongoing" if it hasn't resolved yet, or you simply don't know.

Source link (optional)

A link backing up the report — a news article, an official status page, a company statement.

Attachments (optional)

Screenshots or other files supporting the report, added after the outage is created. The first image becomes the thumbnail shown on the feed unless you mark a different one as primary.

Roles

  • User — the default. Submit outages, comment on published ones, edit/delete your own drafts.
  • Moderator — everything a user can do, plus approve/reject pending submissions and remove comments.
  • Admin — everything a moderator can do, plus manage user accounts from the Admin page: change access level, freeze/unfreeze, reset access, manually verify an email, create accounts directly, delete accounts, and view any user's full submission history.

Email verification (via the magic-link sign-in email) is required before you can submit an outage or post a comment.

Comments

Open on published outages only, for anyone with a verified email. Moderators/admins can remove a comment with a reason (shown to the commenter) and an optional internal note.
